Linux —— 静态函数库和动态加载函数库(共享库)

Linux下有2种函数库：
一种是静态函数库(static libraries)；
另一种叫动态加载函数库(dynamically loaded libraries)，也叫共享函数库(shared libraries)。

1.基本原理和优缺点分析

静态库的代码在编译过程中就已经被载入可执行文件。
缺点：
(1)一般来说可执行文件的文件大小比较大；
(2)如果静态库需要改变，则整个执行文件就需要重新编译，不符合高内聚低耦合的特点；
优点：
(1)”适用性强”，可执行文件不需要外部函数库的支持，复制到其它Linux系统上就可以直接运行；

动态库的代码在可执行文件执行的时候才载入内存，在编译过程只是简单的引用。
缺点：
(1)可执行文件的文件大小比较小；
优点：
(1)提高了代码的复用程度，并降低了程序的耦合度。

2.编译方法

静态库名字一般为libxxx.a，而动态库一般则为libxxx.so。
无论动态库还是静态库，都是由.o文件(object file，中间目标文件，是通过使用gcc将源码.c文件编译而来)创建而来的。

如果编译静态库，则使用ar指令（archiver的缩写）。
把x.o和y.o加入到xy_library.a函数库中。如果xy_library.a不存在则新建一个。

# ar rcs xy_library.a x.o y.o

如果你是使用gcc来编译生成可执行文件，则可以使用-l参数来制定这个函数库。

3.实例分析

本例程共有3个源文件组成：
test.h 函数库的头文件
test.c 函数库的源程序
main.c 用于测试函数库的主程序

它们的代码分别如下：
test.h

void test(void);

test.c

#include <stdio.h> 
void test(void)
{ 
    printf("This is a test!\n"); 
    return;
}

main.c

#include "test.h"
void test(void)
{ 
    test();
    return;
}

将它们编译成可执行文件共有3种方法：

1.编译成多个源文件的方法

分别将test.c和main.c编译成为.o文件：

# gcc -c test.c
# gcc -c main.c

此时当前目录下可以看到生成了test.o和main.c。

接下来将2个.o文件链接为一个叫做fortest的可执行文件：

# gcc test.o main.o -o fortest

可看到当前目录下生成了fortest可执行文件。
运行它~

# ./fortest
This is a test!

至此该种方法演示结束。

2.静态函数库的方法

第一步跟上面的方法1一样，然后使用ar命令将test生成静态函数看libtest.a：

# gcc -c test.c
# ar rcs libtest.a test.o

其中：
r表示在库中插入模板；
c表示创建一个库，不管其是否存在；
s表示写一个目标文件索引到库种，或者更新一个存在的目标文件索引。

最后生成可执行文件fortest：

# gcc -o fortest main.c -static -L. -ltest

其中：
-L.表示要链接的库在当前目录中；
-ltest表示要链接的库叫做libtest.a

测试可执行文件，没啥毛病：

# ./fortest
This is a test!

3.动态函数库的方法

# gcc -shared -fPIC -c test.c
# gcc -shared -fPIC -o libtest.so test.o

# gcc -o fortest main.c -L. -ltest

其中：
同上，-L.表示要链接的库在当前目录中；
-ltest表示要链接的库叫做libtest.so

如果你直接执行./fortest，不好意思，error！

./fortest: error while loading shared libraries: libtest.so: cannot open shared object file: No such file or directory

为什么呢？默认情况下可执行文件是在linux文件系统的/usr/lib等目录下寻找动态加载库文件，
然而我们的libtest.so却是在当前目录下，它理所没法被可执行文件找到。
方法1：将动态库搬移到/usr/lib/目录下

sudo mv libtest.so /usr/lib/

方法2：将LD_LIBRARY_PATH设置为当前目录
由于连接器会默认索索LD_LIBRARY_PATH所指定的目录，我们不妨将当前目录路径赋给它。

# export LD_LIBRARY_PATH=$(pwd)

方法3：ldconfig 目录路径

sudo ldconfig $(pwd)

将当前目录中的动态库可以被系统共享。
实际上是在缓存文件/etc/ld.so.cache中追加了该目录中的共享库。

同上，测试一下可执行文件：

# ./fortest
This is a test!

没毛病~

我通过ls -lh指令查看了这3个可执行文件的大小，好家伙~
使用方法1和3生成的可执行文件约8.5KB，方法2的为792KB。相差了90多倍！！
这更好的印证了静态库的缺点。
这也就是为什么系统中多用动态库的原因了。
